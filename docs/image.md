# Руководство по работе с изображениями

В шаблоне используется компонент Picture для отображения картинок. Нужен он для адаптивности картинок для каждого экрана/формата/плотность пикселей.

> **Особенности**
>
> 1. Стили в основном стоит задавать `img` внутри `Picture` <br />
> 2. onLoad применяется только на `img`, а не на `Picture`
> 3. Все пути к изображениям в `publiс` начинаем со `/`. Даже если без него работает.

**Предупреждение**

> Встроенный компонент `Image` от `next.js` не используем. Он сильно нагружает сервер, поэтому лучше - что бы контент подготавливался заранее

## Решить как скачиваем изображение

В figma пользователь может выгружать изображение в разных видах:

- масштаб экспорта - качество (1x, 2x, 3x, 4x)
- расширение изображения (.png, .jpeg, .svg)

### Масштаб экспорта

В Figma при экспорте изображений указываются значения 1x, 2x, 3x — это множители размера исходного макета.

- 1x — исходный размер макета. Используется для обычных дисплеев.
- 2x — в 2 раза больше. Для экранов с высокой плотностью пикселей `Retina/HiDPI`.
- 3x — в 3 раза больше. Для устройств с очень высокой плотностью пикселей.

_Применение_: позволяет создавать изображения, которые выглядят чётко на любых устройствах без размытости.

> **Особенности**
> Экспорт с использованием множителей обеспечивает чёткость изображения на любых устройствах. Даже на обычных дисплеях (не только `Retina/HiDPI`) масштабирование часто даёт заметно более высокое качество, предотвращая размытость и пикселизацию. <br>
> Поэтому не забываем про 2x, ведь он может дать гораздо лучшее качество, чем 1x

### Расширение изображения

По сути расширение указывает не просто «имя файла», а формат хранения изображения. А формат — это способ сжатия и организации данных.

#### Растровые изображения

Представляют собой набор пикселей, каждый из которых хранит информацию о цвете. Такой формат подходит для фотографий и сложных иллюстраций с множеством оттенков и деталей.

_Недостатком_ является потеря качества при увеличении: при масштабировании картинка становится «зернистой» или размывается

##### Форматы растровых изображений

| Формат   | Итоговый Вердикт                                                                                                  | Объяснение                                                                                                                                                                                                                  |
| -------- | ----------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **PNG**  | Полностью поддерживается на всех платформах, альфа работает                                                       | Универсальный формат с прозрачностью. Без ограничений на iOS, macOS, Windows, Android. Использует сжатие без потерь — идеально для текста, но плохо сжимает фото (файл крупнее, чем у JPEG).                                |
| **JPEG** | Полностью поддерживается, **без альфы**                                                                           | Широко используемый формат. Не поддерживает прозрачность. Применяет сжатие с потерями — отлично уменьшает размер фотографий и градиентов (обычно в 5–10 раз меньше PNG), но может давать артефакты при низком качестве.     |
| **WebP** | Поддерживается на всех современных платформах; на старых iOS до 14 — нет поддержки альфы. Есть поддержка анимации | WebP умеет прозрачность (WebP α), но Safari до версии 14 её не рендерит. Chrome/FF/Edge поддерживают полностью. Имеет более эффективное сжатие анимации, чем gif, поэтому для не старых клиентов является отличной заменой. |
| **AVIF** | Поддерживается на современных платформах; на старых iOS/macOS ограничено. Есть поддержка анимации                 | AVIF поддерживает альфа-канал, HDR и высокий битрейт, но Safari до 16.3 не поддерживает, старые Android и Windows старые версии — частично. Обычно даёт файлы в 2–3 раза меньше WebP при том же качестве.                   |
| **GIF**  | Работает везде; альфа (прозрачный цвет) поддерживается; анимация — да                                             | Старый формат, поддерживает прозрачность только как однократный цвет (1-bit α), не true alpha.                                                                                                                              |

##### Уточнение между jpeg и png

| Характеристика             | **PNG**                                                   | **JPEG**                                                |
| -------------------------- | --------------------------------------------------------- | ------------------------------------------------------- |
| **Тип сжатия**             | Без потерь (lossless)                                     | С потерями (lossy), настраиваемое                       |
| **Размер файла**           | ❌ Обычно больше, особенно для сложных изображений        | ✅ Меньше, особенно при высоком уровне сжатия           |
| **Типы изображений**       | ✅ скриншоты, фотографии и изображения с прозрачным фоном | ✅ Фотографии, изображения с плавными переходами цветов |
| **Поддержка альфа-канала** | ✅ Да                                                     | ❌ Нет                                                  |
| **Прогрессивная загрузка** | ❌ Нет                                                    | ✅ Да (изображение загружается постепенно)              |

Если нужно прозрачность, чёткость или без потерь — PNG; если это фотография без прозрачности и важен малый размер — JPEG.

#### Векторные изображения (SVG)

Cтроятся из математически описанных примитивов — точек, линий, кривых и многоугольников. Благодаря этому они сохраняют чёткость при любом увеличении и идеально подходят для логотипов, иконок и схем.

_Ограничение_ в том, что векторные форматы плохо передают плавные цветовые переходы и фотореализм.

##### Добавление SVG

Для добавления SVG на страницу в шаблоне можно использовать 3 метода:

1. Использовать `<img>`, указав в `src`

> **Внимание:**
>
> 1. При таком способе svg не теряет своих векторных свойств (не становится растровым) <br />
> 2. По сравнению со 2 способом имеет большие преимущества, поэтому способ оказывается предпочтительнее

2. Использовать импорт svg напрямую

```tsx
import LogoSvg from "@/public/logo.svg";
```

> **Внимание**
> При использовании такого метода:
>
> 1. svg попадает в bundle js, что может увеличить размер javascript кода
> 2. браузер не кеширует такое svg как отдельный файл
> 3. inline SVG воспринимается иначе, чем `<img>` с alt. Придётся добавлять title/aria-label вручную.

Такой способ может помочь лучше кастомизировать svg (например его внутренние элементы)

3. Вставить svg как есть - скопировать исходный код svg (как React компонент)

Этот способ используется в исключительных случаях, когда нам нужно сильно кастомизировать svg. Например: убирать / добавлять отдельные части, навешивать атрибуты на разные элементы svg и т.д

##### fake svg

Растровое изображение (PNG, JPG, WebP), закодированное в base64, вставленное внутрь SVG через тег <image>.

Внешне выглядит как SVG, но внутри — обычный растр.

> **Наблюдение**
> Очень часто у такой иконки на макете можно заметить `background: url(<path-to-image>)`

Для SVG следует использовать исключительно векторную графику. Растровые изображения вставлять в SVG не рекомендуется, так как это снижает эффективность и увеличивает размер файла. Для обычных картинок лучше использовать элементы <img> или <picture>.

## Обработка изображения на примере

### Выбор стандартов

Стандартные форматы:

- 1x desktop
- 2x desktop

- 1x tablet
- 2x tablet

- 2x mobile

### Уменьшение размера изображения

Есть 2 пути:

- сжатие
- конвертирование

Лучший pipeline для изображения:

1. Оригинальное изображение сжимаем - это изображение мы используем в качестве fallback изображения - так как имеет формат, который поддерживается всеми пользователями

2.1. Оригинальное изображение конвертируем - в частности это формат webp. (Можно еще avif)
2.2. Сконвертированное оригинальное изображение в webp - сжимаем

Это изображение будет использоваться в качестве оптимизированного решения, который поддерживается не всеми пользователями, но более 90% из них

> **Наблюдение**
> Почему нельзя сжать оригинальное изображение и конвертировать в webp? <br />
> Потому что при конвертации в webp - такое изображение может увеличиваться в размере относительно
> сжатого и оно никогда не будет меньше, чем оригинальное изображение, сконвертированное в webp

### Итоговый набор

png/jpeg 2x desktop - сжатое <br>
png/jpeg 1x desktop - сжатое <br>

webp 2x desktop - сжатое (по возможности) <br>
webp 1x desktop - сжатое (по возможности) <br>

png/jpeg 2x tablet - сжатое <br>
png/jpeg 1x tablet - сжатое <br>

webp 2x tablet - сжатое (по возможности) <br>
webp 1x tablet - сжатое (по возможности) <br>

png/jpeg 2x mobile - сжатое <br>
webp 2x mobile - сжатое (по возможности) <br>

## Отслеживание загрузки изображения

В шаблоне есть хук `useImageReady` - сделан он для удобного управления готовностью изображения.

Особенность этого процесса в том, что события для проверки готовности картинки не вызываются при кешированной картинке. В кешированной картинке устанавливается сразу `readyState`, поэтому использовать только события - **не получится** - нужно проверять готовность состояния картинки, что и происходит в этом хуке:

```tsx
if (image.complete) {
  // ...
}
```

## Финал

В итоге: для современной графики используем `WebP` и `AVIF` через `<picture>`, а в качестве `fallback` оставляем `PNG` и `JPG` (в зависимости от наличия альфа-канала), чтобы покрыть все браузеры и сценарии.

Для логотипов и иконок используем `SVG`
