# Краткое введение

IDP — это экосистема инструментов и сервисов, которая поддерживает весь процесс разработки программ внутри компании. Её можно сравнить с единым окном, через которое команды самостоятельно получают ресурсы: серверы, тестовые среды, базы данных и другие инструменты.

Цель внедрения Internal Developer Portal (IDP) — централизовать инструменты разработки и автоматизировать процессы во благо скорости и качества разработки. Основные задачи включают ускорение онбординга новых разработчиков, повышение автономности команд, автоматизацию DevOps-операций, улучшение видимости состояния систем (мониторинг, метрики, оповещения) и обеспечение соответствия внутренним стандартам безопасности и качества. Такие порталы уже в процессе дают «быстрый онбординг, снижение нагрузки на DevOps и большую согласованность/надёжность сервисов»

# Ключевые компоненты IDP

В основе IDP лежит не набор конкретных инструментов, а система взаимосвязей и автоматизации. Главная ценность платформы заключается в её способности объединять разрозненные элементы инфраструктуры в целостную экосистему с общими API, стандартами и интерфейсом. Платформа благодаря этому превращает все этапы жизненного цикла программного обеспечения в единый понятный процесс.

При этом конкретная реализация, — будь то Git, Jira или другие продукты, — вторична по отношению к выстроенным процессам. Важны не сами инструменты, а то, как они связаны между собой и насколько эффективно взаимодействуют.

Ключевая задача IDP — автоматически находить и выстраивать эти связи, обеспечивая прозрачность и эффективность на каждом этапе. Платформа предоставляет инженерам не просто коллекцию разрозненных инструментов, а целостную среду, где все компоненты гармонично соединены от начала до конца цикла разработки. Это позволяет командам получать нужные ресурсы в нужный момент без лишних сложностей и задержек.

# Подробные требования к системе Internal Developer Portal (IDP)

Ниже — исчерпывающая спецификация требований к внутреннему порталу разработчиков (IDP). Раздел охватывает цели, функциональные и нефункциональные требования, сценарии использования и приёмочные критерии, модель данных, API, вопросы безопасности, метрики и дорожную карту по фазам (MVP → расширение).

---

## 1. Цель и задачи системы

**Цель:** предоставить единый, безопасный и удобный «вход» для разработчиков и команд, чтобы ускорить разработку, упростить онбординг, стандартизировать декларирование сервисов и интегрировать инструменты разработки (CI/CD, observability, secrets, инфраструктура).

**Ключевые задачи:**

- Каталог сервисов и компонентов (service catalog).
- Быстрая и стандартизированная генерация новых проектов (scaffolding/templates).
- Интеграция с CI/CD, артефактными репозиториями, системами мониторинга и логов.
- Единый интерфейс для доступа к документации, runbooks, ownership и контрактам.
- Управление доступом (RBAC/ABAC), аудит действий и соответствие политикам безопасности.
- Метрики эффективности платформы (onboarding time, активность команд, health of services).
- Автоматическая генерация документации проектов с использованием искусственного интеллекта
- Интеграция с Jira
- Интеграция с Gitlab/Github/BitBucket
- Встроенные графические инструменты проектирования
- Аудит безопасности и уязвимостей (ai)
- Генерация самммари проекта за день
- Кастомизация страницы виджетами

## 2. Заинтересованные стороны

- Разработчики (frontend/backend, мобильные)
- Платформенная/DevOps команда
- Руководители инженерных команд
- SRE[^1]/операционные команды
- Product- и техлиды
- Security/Compliance

---

## 3. Область применения

**Включено:**

- Централизованный каталог сервисов/компонентов
- Scaffolder/шаблоны проектов 
- Интеграции: SCM (GitLab/GitHub/Bitbucket), CI/CD (Jenkins/GitHub Actions/GitLab CI), Artifact Registry, Docker Registry, Vault/Secrets Manager, Monitoring (Prometheus/Grafana), Tracing (Jaeger), Logging (ELK/EFK)
- Single Sign-On (OIDC/SAML) + SCIM provisioning
- RBAC/Teams/Ownership
- Audit logging, alerting, SLO/KPI dashboard
- REST/GraphQL API и Webhooks
- Multi-environment support (dev/stage/prod), multi-cluster/multi-cloud

**Исключено (не в MVP):**

- Генерация инфраструктуры (IaC) «из коробки» без интеграции с существующими инструментами
- Платёжные/биллинг-инструменты (если не требуется)
- Полностью автоматическое remediation (можно как расширение)

---
## 4. Пользовательские роли и права

**Разработчики:** используют портал как единую точку входа для поиска информации о сервисах (каталог сервисов), просмотра документации и API (TechDocs), создания новых проектов по шаблонам (Scaffolder/Software Templates) и запуска self-service операций (развертывание сервисов, управление секретами и др.). Разработчики оценивают готовность сервисов по метрикам качества, SLA/SLO и состоянию инцидентов.
    
**DevOps-инженеры / Platform-специалисты:** отвечают за настройку и поддержку IDP: определяют шаблоны проектов, настраивают интеграции с CI/CD и системами мониторинга, устанавливают политики безопасности и аудита (SSO, RBAC). Они автоматизируют provision инфраструктуры и внедряют стандарты (scorecards, compliance checks), обеспечивая самодостаточность команд.
    
**Тестировщики (QA):** получают доступ к тестовым окружениям через портал, следят за соблюдением стандартов качества (покрытие тестами, метрики стабильности, соблюдение SLO) и могут инициировать или отслеживать тестовые случаи.
    
**Менеджеры проектов / Engineering Managers:** используют IDP для обзора статуса разработки и эксплуатации (дашборды, отчёты по показателям сервисов), контроля выполнения стандартов и SLA/SLO, координации межкомандных изменений (кампании/инициативы) и планирования ресурсов.

### Черновой вариант реальных ролей
- Admin - имеет полные права в проекте
- Member - член проекта, базовые права чтения

## 6. Функциональные требования

### 6.1 Каталог сервисов

- Хранение метаданных сервисов: имя, описание, owners (team/person), связные репозитории, окружения, SLOs, health status, dependencies, contacts, tags, SLA, docs URL.
- UI: страница сервиса с карточкой, история изменений, quick actions (open repo, open pipeline, view logs, open dashboard).
- Поддержка версионирования манифестов сервиса.
- Импорт из внешних источников (Backstage-like, Kubernetes, Git repo scan).
- API: CRUD для сервисов + bulk import/export.


**Приёмочные критерии:**

- Можно зарегистрировать сервис с полным набором метаданных.
- Для каждого сервиса доступны быстрые действия — repo, pipeline, metrics, logs.
- Поиск возвращает релевантные сервисы по имени/тегам/owner.


### 6.2 Управление доступом и безопасность

Единая аутентификация (SSO через Okta/AD или OIDC) и тонкая система ролей (RBAC) с аудитом действий. Специалисты подчеркивают необходимость «надёжной аутентификации, контроля доступа (RBAC/SSO) и аудита операций». Портал должен интегрироваться с корпоративной системой идентификации для централизованного управления правами.

### 6.3 Документация и API (TechDocs)

Внутренняя техническая документация (docs-as-code) и спецификации API. Backstage, например, предоставляет плагин TechDocs для генерации документации из репозиториев. Портал объединяет информацию по API и документацию для удобства навигации и поддержки.

### 6.4 Поиск и навигация

- Быстрый полнотекстовый поиск по названию, описанию, тегам, owners.
- Фильтры: language, team, environment, status, tags.
- Сортировка по активности, времени обновления, owner.

### 6.5 Scaffolder / Templates

- Поддержка шаблонов (project/component templates) — параметры при создании.
- Генерация репозитория с recommended CI pipeline, IaC skeleton, Dockerfile, initial README, OWNERS.
- Возможность установки policy checks в шаблоне (security lint, license).
- UI-формы для заполнения параметров шаблона; preview изменений перед создание PR.
    

**Приёмочные критерии:**

- Разработчик может создать новый проект из шаблона; генерируется PR в выбранный git-репозиторий.
- Шаблоны доступны по ролям/командам.

### 6.6 Интеграция с SCM и CI/CD

- Поддержка OAuth/OIDC/Deploy keys и webhooks для GitHub/GitLab/Bitbucket.
- Отображение статусов pipeline, тестов, артефактов.
- Возможность запуска pipeline прямо из UI (ограничения RBAC).
- Поддержка отображения artefact links (image tags, versions).

Портал агрегирует результаты билдов и деплоев и может инициировать пайплайны «из своего интерфейса». Так, Port может запускать задачи в Jenkins и GitHub Actions через узлы Blueprint[^6]

**Приёмочные критерии:**

- Статус pipeline отображается на странице сервиса в реальном времени (или с минимальной задержкой).
- Можно вызвать job (если разрешено) и отслеживать его лог в UI.

### 6.7 Обнаружение зависимостей и сервисная карта

- Визуализация зависимостей между сервисами (graph), включая вызовы HTTP/gRPC, топики и подписки (Kafka).
- Автоматическое обновление графа на основе tracing/ingress/registrations.

### 6.8 Документация и Runbooks

- Хранилище документации (Markdown), привязка к сервисам.
- Runbooks с шаблонами шагов для инцидентов, checklists.
- Возможность redline/предлагать правки (PR workflow) или in-place editing с версионированием.

### 6.9 Secrets & Configuration (интеграция)

- Не хранить секреcты напрямую. Поддержка ссылок/рендеров к Vault/Secrets Manager.
- UI для просмотра метаданных секрета (без раскрытия значений), управление access policies через platform admin.
- Возможность инициировать ротацию секрета (через интеграцию).

### 6.10 Notifications & Alerts

- Поддержка каналов: email, Slack/MS Teams/Telegram, webhook.
- Настройка таргетинга: owner/team/role/alert type.

### 6.11 Audit, Logs и Compliance

Сбор и отображение метрик производительности, надёжности и инцидентов сервисов. Многие IDP включают встроенные дашборды: Cortex, например, предоставляет «метрики состояния сервисов» и таргетированные оповещения для инженеров.

Портал должен интегрироваться с внешними системами мониторинга (Prometheus, Datadog, Grafana и др.) для единого обзора состояния и SLA/SLO.

- Все критические действия (create/update/delete service, change RBAC, run scaffolder) — логируются с immutable storage.
- Audit export в CSV/JSON, интеграция с SIEM (Splunk/ELK).
- Retention policy: configurable (например, 1 year default).

### 6.12 API & Webhooks

- Public API (REST/GraphQL) для интеграции других инструментов.
- Webhooks для событий: service.created, service.updated, pipeline.status, secret.rotated, template.used.
- Throttling и access tokens: per-app tokens, scoped.

---

## 7. Нефункциональные требования

### 7.1 Надёжность и доступность

- Цель: availability target (SLA) — например, 99.9% (SLO: 99.95% для критичных функций read-only).
- Поддержка отказоустойчивых развертываний (multi-AZ, multi-cluster).
    
### 7.2 Масштабируемость

- Горизонтальная масштабируемость frontend/API/server components.
- Архитектура должна выдерживать: N активных пользователей (настраивается) и M зарегистрированных сервисов (десятки/сотни/тысячи).

### 7.3 Безопасность

- SSO (OIDC/SAML) + MFA enforced for privileged roles.
- Шифрование: TLS in transit; at-rest encryption for баз данных и объектов.
- Secrets values never stored in plain text; use external secrets managers.
- RBAC fine-grained; support SCIM provisioning for user sync.
- Regular security scans, dependency checks, pen-tests annually or on major changes.

### 7.4 Логирование и мониторинг

- Centralized metrics (Prometheus format), distributed tracing, structured logs (JSON).
- Alerting on health-check failures, queue/backlog growth, error rate anomalies.

---

## 8. Мониторинг и метрики (предложение)

- Platform health: uptime, api_latency_p95, api_errors_5xx_rate.
- User metrics: active_users_daily, new_projects_weekly, average_onboarding_time.
- Service metrics: number_of_services, services_with_slo_violations, percent_services_with_docs.
- Operational: webhook_failures, queue_backlog, template_instantiate_failures.

Рекомендуется экспорт метрик в Prometheus + Grafana дашборды и alerting через Alertmanager.

---

## 9. Тестирование и качество

- Unit tests для backend, e2e для основных user flows (create service, instantiate template, view pipeline).
- Security tests: SAST/DAST, dependency scanning, container image scanning.    
- Performance tests: load test APIs for search and catalog endpoints.
- Accessibility testing (automated + manual spot checks).
    

---

## 10. Безопасность и соответствие

- SSO (OIDC or SAML) + enforce MFA for privileged roles.
- SCIM для provisioning пользователей/teams.
- Encryption: TLS everywhere; DB + object storage encrypted at rest.
- Secrets never visible in UI; only references and metadata.
- Audit logs immutable (WORM or external SIEM).
- Periodic pen-test, dependency vulnerability patching cadence.
    

---
## 11. Требования к архитектуре и интеграциям
### Архитектура
Контейнеризированная, микросервисная реализация (например, Kubernetes) для масштабирования и отказоустойчивости. Backstage, например, оптимизирован под Kubernetes и использует API-first подход. Система должна выдерживать тысячи и даже сотни тысяч записей в каталоге: Cortex способен «обрабатывать сотни тысяч сущностей» при встроенном контроле rate-limits и кэшировании[^2]. Рекомендуется модульная (плагинная) архитектура для лёгкого расширения (как у Backstage).
    
### Интеграции
Поддержка основных инструментов: 
1. Системы контроля версий (GitHub, GitLab, Bitbucket); 
2. CI/CD (Jenkins, GitHub Actions, GitLab CI, Azure DevOps и др.); 
3. Облачные платформы (AWS, GCP, Azure); 
4. Системы мониторинга и логирования (Prometheus, Datadog, Grafana); системы трекинга инцидентов (Jira, PagerDuty); 
5. Менеджеры секретов (Vault, AWS Secrets Manager) и системы идентификации (Okta/LDAP). Cortex, например, заявляет поддержку 50+ встроенных интеграций (OpsLevel – webhooks)[^3]. При высокой нагрузке важно контролировать rate-limits: Cortex автоматически обрабатывает лимиты[^2], в то время как в других решениях (OpsLevel) для крупных систем требуется ручной шардинг вебхуков.

### Безопасность
Обязательна интеграция с корпоративной системой аутентификации (SSO/SAML/OIDC, Okta, LDAP). Портал должен вести аудит всех операций, обеспечивать RBAC на уровне сущностей и соответствовать стандартам безопасности (SOC 2, ISO, GDPR и др.). Доступ к секретам и конфигурациям должен быть надёжно зашифрован и ограничен.
    
### Хостинг и надёжность
Возможность развернуть как облачное (SaaS), так и on-premise (контейнеры заказчика) решение для соответствия разным требованиям безопасности. Платформа и её компоненты должны поддерживать автоматические обновления (rolling-updates), резервное копирование данных, а также гибкое управление версиями.
    
### Производительность
Портал должен быстро отвечать на запросы каталога и поиска. Для масштабируемости критично распределение нагрузки (горизонтальное масштабирование) и кеширование. При превышении квот API важно их корректно обрабатывать и троттлить.
    
### Управление данными
Конфигурации портала, шаблоны и политики должны храниться в системах контроля версий (GitOps). Желательна периодическая верификация актуальности данных (entity verification)[^4], чтобы выявлять неактуальные или «заброшенные» сервисы. Поддержка иерархий (как оргструктур или продуктовых линий) должна быть гибкой (Cortex позволяет рекурсивные связи, OpsLevel – базовые)[^5]

## 12. Roadmap (фазы, без временных оценок)

**MVP (обязательное):**

- Service catalog CRUD + UI.
- Basic search.
- Scaffolder (1–2 шаблона).
- SCM integration (read statuses, webhooks).
- SSO + RBAC.
- Audit logging.
- Public REST API.
- Basic metrics & health dashboards.

**Phase 1 (расширение):**

- CI/CD triggers + run pipelines from UI.
- Secrets manager integration (reference-only).
- Policy-as-code integration (basic checks).
- Template marketplace + template approval workflow.
- Notifications (Slack/Teams).

**Phase 2 (продвинутые возможности):**

- Dependency graph visualization (auto-discovery).
- Cost insights per service (если данные доступны).
- Multi-tenant/team isolation.
- Advanced governance: auto-remediation workflows, change approval flows.
- Plugin system / extensibility marketplace.

## 13. UX / UI рекомендации

- Card-based design для сервисов с quick actions.
- На странице сервиса — один экран с ключевой информацией: owners, repo, pipeline, metrics, runbooks, tags.
- Формы шаблонов с валидацией полей и preview PR diffs.
- Удобная on-boarding flow: «Create first service» wizard с подсказками.

## 14. Анализ конкурентов

### Backstage (Spotify, OSS)

Фреймворк с открытым исходным кодом (CNCF-проект) для создания IDP. Включает каталог сервисов, шаблоны проектов (Scaffolder) и документацию (TechDocs). 

Плюсы – высокая гибкость и расширяемость через плагины, отсутствие лицензионных затрат. 

Минусы – длительное внедрение: по опыту ряда компаний это 6–12 месяцев и десятки человеко-месяцев на поддержку; низкий процент активного использования (~10% разработчиков) из-за сложности и избытка информации[^7].
    
### Port

Коммерческий SaaS-IDP с удобным визуальным редактором процессов («Blueprints»). Предоставляет настраиваемый каталог с богатыми связями между сущностями и мощными self-service действиями: через встроенные интеграции можно запускать задания в GitHub Actions, Jenkins и др. Имеются готовые scorecards для оценки соответствия стандартам. 

Плюсы – быстрое развертывание (POC настраиваются за дни), минимальный overhead (SaaS), интуитивный UI.

Минусы – сложность настройки Blueprint-сценариев, ручной уход за каталогом при слабой интеграции, а также высокая стоимость при масштабе (enterprise-план вдвое дороже конкурентов).
    
### Cortex

SaaS-IDP «корпоративного класса» с упором на метрики качества и стандарты разработки. Cortex сочетает усиленный каталог (интеграция с Git, CI/CD, инцидент-трекерами) и расширенные scorecards для покрытия тестами, SLO и проверок безопасности. Портал предоставляет метрики здоровья сервисов и отчёты для руководства. Плюсы – готовый стек практик и аналитики «из коробки», AI-подсказки (автоматическое назначение владельцев сервисов с высокой точностью.

Минусы – жёсткая модель данных, длительное внедрение (несколько месяцев) и высокая стоимость (~$65–69/польз./мес).
    
### OpsLevel

SaaS-IDP, позиционирующийся как готовое решение «из коробки» для каталога и контроля качества. Автоматически заполняет каталог из Git-репозиториев, Kubernetes-кластеров и облачных аккаунтов. Есть шаблоны сервисов, одношаговые действия (Actions) и механизмы кампаний (Campaigns) для массовых изменений, а также настраиваемые scorecards (Checks) для соответствия требованиям.

Плюсы – быстрое внедрение (30–45 дней), простота использования, обширные готовые интеграции. 

Минусы – ограниченная гибкость (Actions без сложных сценариев, единые правила проверок), проприетарность (нельзя менять ядро платформы); стоимость около $39/польз./мес.
    
### Atlassian Compass

SaaS-IDP, тесно интегрированный с Jira, Confluence и Bitbucket. Предоставляет каталог компонентов, связанный с документацией, задачами и репозиториями, плюс scorecards для оценки «здоровья» компонентов. 

Плюсы – простота старта (есть бесплатный план до 3 пользователей и недорогие тарифы), глубокая интеграция с Atlassian-экосистемой. 

Минусы – фокус на Atlassian-инструментах (мало интеграций вне Jira/Confluence/Bitbucket), упрощённый функционал без сложных автоматизаций и невысокая кастомизация.

## 15. Сравнительный анализ IDP-платформ

| **Платформа** | **Модель**            | **Основные возможности**                                                                                                             | **Примечания**                                                                                                                                                          |
| ------------- | --------------------- | ------------------------------------------------------------------------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Backstage** | Open Source (on-prem) | Каталог сервисов, TechDocs, шаблоны проектов; расширяемость через плагины.                                                           | Максимальная гибкость, нет лицензионных расходов, нет vendor-lock-in. Требует много ресурсов на внедрение (6–12 мес, 3–15 FTE).                                         |
| **Port**      | SaaS                  | Гибкий софткаталог с визуальными Blueprint-процессами; self-service actions (запуск CI/CD через GitHub Actions, Jenkins).            | Быстрый старт и удобный UI; готовые scorecards; SaaS-поддержка. Однако сложная поддержка Blueprint-сценариев и высокие затраты при росте (enterprise-план вдвое дороже) |
| **Cortex**    | SaaS                  | Корпоративный каталог (интеграции с Git, CI/CD, инцидентами); продвинутые scorecards (тесты, SLO, безопасность); аналитика сервисов. | Ориентирован на крупные команды; AI-подсказки (владельцы сервисов). Жёсткая модель данных, долгий TTM (6+ мес) и высокая стоимость (~$65–69/мес)                        |
| **OpsLevel**  | SaaS (on-prem опция)  | Автоматический каталог (Git/K8s/облако); шаблоны сервисов, одношаговые Actions, кампании; чек-листы (Checks)                         | Быстрое развёртывание (30–45 дней), простота и готовые интеграции; ≈$39/мес на пользователя. Меньше гибкости (Actions без сложных сценариев, единые правила проверок).  |
| **Compass**   | SaaS (Atlassian)      | Каталог компонентов, tight интеграция с Jira/Confluence/Bitbucket; scorecards для метрик качества                                    | Очень просто начать (есть бесплатный план до 3 пользователей); ограничена экосистемой Atlassian, упрощённый функционал и малая кастомизация.                            |
