# Code Style Guide

## 1. Структура проекта

```
src/
  app/                # Страницы и layout'ы Next.js
  shared/             # Общие компоненты, утилиты, типы, константы, стили, хуки, API
    ui/               # Атомарные и молекулярные компоненты (каждый в своей папке)
    lib/              # Утилиты и хелперы
    types/            # Типы и интерфейсы
    constants/        # Константы
    styles/           # Общие стили
    hooks/            # Кастомные хуки
    api/              # Общие API-запросы
  entities/           # Бизнес-сущности (каждая в своей папке)
  features/           # Фичи (каждая в своей папке)
  widgets/            # Виджеты (каждый в своей папке)
  views/              # Крупные страницы (каждая в своей папке)
  config/             # Конфиги, глобальные настройки
  public/             # Статика
```

---

## 2. Именование файлов и папок

- **Папки:** lower-kebab-case (`card/`, `policy/`, `attach-file/`)
- **Файлы компонентов:**
  - `index.tsx` — основной компонент
  - `*.module.css` — стили компонента
  - Для типов: `types.ts`
- **Утилиты, константы, хуки:** kebab-case (`check-is-safari.ts`)
- **Картинки и другие статические дынные:** kebab-case (`team-icon.png`)

---

## 3. Именование переменных, функций, классов

- **Компоненты:** PascalCase (`Card`, `PolicyPage`)
- **Типы/интерфейсы:** PascalCase с припиской Type / Props (`CalendarStateType`, `CalendarTeamProps`)
- **Перечисления** PascalCase с припиской Enum на конце (`UserRoleEnum`, `DimensionEnum`)
- **Функции/утилиты:** lowerCamelCase (`getSitemapObject`, `checkIsSafari`)
- **Константы:** UPPER_SNAKE_CASE (`CALENDAR_CARD_TYPE`, `ACTIVE_CLASS`)
- **CSS-классы:** lowerCamelCase (`root`, `poster`)
- **CSS-переменные** kebab-case (`--width-video`, `--color-blue`)
- **Обработчики событий:** с префиксом `handle` (`handleClick`)
- **Свойства объекта** lowerCamelCase (`isPolicy`)

---

## 4. Стиль кода

- **Типизация:**
  - Всегда использовать строгую типизацию (TypeScript)
  - Для props компонентов — отдельный тип с суффиксом `Props`. Находится рядом с компонентом
  - Для данных/ответов API — отдельные типы с суффиксом `Type`
  - Типизация пропсов компонентов внутри аргумента функции `({ ...props }: FunctionProps)`
- **Экспорт:**
  - Экспорт компонентов и функций через `export const`
  - Экспорт типов через `export type`
- **Комментарии:**
  - В начале каждого файла — блок copyright
  - JSDoc используется по необходимости, в основном краткие комментарии
- **JSX:**
  - Явный возврат (`return (...)`)
  - Классы через `clsx`/`cx`
  - props компонента — деструктуризация в сигнатуре
  - Проверка данных на пустоту должна быть на уровне компонента, чтобы у нас не было пустых тегов в итоговой верстке
  ```
  {description && (
  				<p className={css.description}>{parser(description)}</p>
  			)}
  ```
- **CSS:**
  - Только CSS-модули
  - Для отступов используем только логические свойства. Вместо `margin-top` -> `margin-block-start` и т.д
  - Модули находятся рядом с `.tsx` файлом компонента
  - Именуется так `import css from './index.module.css'`
  - Классы — короткие, осмысленные
  - Медиа-запросы — внутри файла компонента

## 5. Текст

- Весь текст выносим в константы. Обычно - это константа страницы / секции.
- Весь текст прогоняем через типограф (союзы для/или не учитываются, но мы учитываем)
- Типограф - https://www.artlebedev.ru/typograf/

## 6. Примеры оформления для сущностей (абстрактно)

### 6.1. Feature (фича)

- Каждая фича — отдельная папка в `features/`.
- Внутри: основной компонент (`index.tsx`), вспомогательные компоненты, утилиты, стили, типы.
- Название папки и файла — по смыслу фичи, lower-kebab-case.
- Экспортируется только основной публичный компонент/функция.
- Типы props и внутренние типы — отдельным файлом или внутри компонента.

### 6.2. Entity (сущность)

- Каждая сущность — отдельная папка в `entities/`.
- Внутри: основной компонент (`index.tsx`), UI-части, типы, стили, helpers.
- Название папки и файла — по смыслу сущности, lower-kebab-case.
- Экспортируется только основной публичный компонент/функция.
- Типы props и внутренние типы — отдельным файлом или внутри компонента.

### 6.3. Widget (виджет)

- Каждый виджет — отдельная папка в `widgets/`.
- Внутри: основной компонент (`index.tsx`), вспомогательные компоненты, модель, стили, типы.
- Название папки и файла — по смыслу виджета, lower-kebab-case.
- Экспортируется только основной публичный компонент/функция.
- Типы props и внутренние типы — отдельным файлом или внутри компонента.

### 6.4. API (запросы)

- Каждый запрос — отдельный файл в `shared/api/`.
- Название файла — по смыслу запроса, kebab-case (`get-events.ts`).
- Внутри: функция запроса, константы, типы ответа.
- Экспортируется только функция запроса и необходимые константы/типы.

---

## 7. Прочее

- **DRY:** не дублировать код, выносить повторяющееся в shared/
- **Доступность:** использовать aria-атрибуты, tabindex, обработчики клавиатуры для интерактивных элементов
- **Порядок props:** сначала обязательные, потом опциональные, потом children
- **Порядок функций:** сначала хелперы, потом основной компонент
- **Асинхронность:** для страниц Next.js — асинхронные компоненты (async/await)
